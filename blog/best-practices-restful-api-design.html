Best Practices for RESTful API Design

May 22, 2023 6 min read Shahorior
API REST Node.js
Share:

Designing a good API is more art than science. A well-designed API can significantly improve developer productivity, code quality, and user satisfaction. Whether you're building an API for internal use, partners, or public consumption, following these RESTful API design best practices will help ensure your API is intuitive, efficient, and maintainable.

Use Nouns, Not Verbs in Endpoint Paths

REST APIs use HTTP methods to indicate actions, so your endpoint URLs should focus on the resources (nouns) rather than the actions (verbs). This approach makes your API more intuitive and easier to understand. 


/* Good */
GET /articles             # Get all articles
GET /articles/123         # Get a specific article
POST /articles            # Create a new article
PUT /articles/123         # Update article 123
DELETE /articles/123      # Delete article 123

/* Bad - using verbs in URLs */
GET /getArticles
GET /getArticle/123
POST /createArticle
PUT /updateArticle/123
DELETE /deleteArticle/123

Use HTTP Methods Correctly

RESTful APIs should use HTTP methods according to their defined purpose:

  • GET: Retrieve a resource or collection of resources (read-only)
  • POST: Create a new resource
  • PUT: Update a resource by replacing it entirely
  • PATCH: Partially update a resource
  • DELETE: Remove a resource

Making proper use of these methods ensures your API follows standard conventions, making it more intuitive for developers.

Use Proper HTTP Status Codes

HTTP response status codes provide valuable information about the result of a request. Using them correctly makes your API more self-descriptive and easier to debug. 


// Success responses
200 OK              // Standard success response
201 Created         // Resource created successfully
204 No Content      // Success but no content to return (e.g., for DELETE)

// Client error responses
400 Bad Request     // Invalid request format
401 Unauthorized    // Authentication required
403 Forbidden       // Authentication succeeded but user lacks permissions
404 Not Found       // Resource not found
422 Unprocessable Entity  // Validation errors

// Server error responses
500 Internal Server Error  // Something went wrong on the server
503 Service Unavailable    // Server temporarily unavailable

Use Plural Nouns for Collection Resources

For consistency and clarity, use plural nouns for collections and allow accessing specific items by ID: 


GET /articles              # Get all articles
GET /articles/123          # Get article with ID 123
GET /articles/123/comments # Get all comments for article 123

Nest Resources for Relationships

When resources are related, design endpoints that reflect these relationships. However, don't nest resources too deeply as it can make the API harder to use. 


/* Good - One level of nesting */
GET /articles/123/comments        # Get all comments for article 123
POST /articles/123/comments       # Add a comment to article 123

/* Avoid - Too deep nesting */
GET /articles/123/comments/456/replies/789  # Too many levels

For deeply nested resources, consider using query parameters or creating separate endpoints.

Use Consistent Naming Conventions

Consistency is key to a good API design. Choose a naming convention and stick to it throughout your API:

  • Use kebab-case or snake_case for URLs (e.g., /blog-posts or /blog_posts)
  • Use camelCase or snake_case for query parameters (e.g., ?sortBy=createdAt or ?sort_by=created_at)
  • Use camelCase or snake_case for JSON property names, matching your language conventions

Implement Proper Filtering, Sorting, and Pagination

For collections that might return many items, provide mechanisms for filtering, sorting, and pagination through query parameters: 


# Filtering
GET /articles?status=published
GET /articles?author=john
GET /articles?category=technology

# Sorting
GET /articles?sort=created_at
GET /articles?sort=-created_at    # Descending order with the minus prefix

# Pagination
GET /articles?limit=10&offset=20
GET /articles?page=2&per_page=15

Include metadata about pagination in your responses: 


{
  "data": [...],
  "meta": {
    "total": 50,
    "per_page": 10,
    "current_page": 2,
    "last_page": 5,
    "from": 11,
    "to": 20
  },
  "links": {
    "self": "/articles?page=2",
    "first": "/articles?page=1",
    "prev": "/articles?page=1",
    "next": "/articles?page=3",
    "last": "/articles?page=5"
  }
}

Versioning Your API

APIs evolve over time. To prevent breaking changes from affecting existing clients, implement versioning. There are several approaches: 


# URI path versioning
GET /v1/articles
GET /v2/articles

# Query parameter versioning
GET /articles?version=1
GET /articles?version=2

# Custom header versioning
GET /articles
API-Version: 1

# Accept header versioning
GET /articles
Accept: application/vnd.mycompany.v1+json

URI path versioning is most common as it's the simplest to understand and implement.

Use Proper Error Handling

Well-structured error responses help developers debug issues. Include helpful error messages, error codes, and potentially links to documentation. 


{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "The request could not be processed due to validation errors",
    "status": 422,
    "details": [
      {
        "field": "email",
        "message": "Must be a valid email address"
      },
      {
        "field": "password",
        "message": "Must be at least 8 characters long"
      }
    ],
    "documentation_url": "https://api.example.com/docs/errors/validation"
  }
}

Implement HATEOAS for Discoverability

HATEOAS (Hypermedia as the Engine of Application State) enhances API discoverability by including relevant links in responses. This helps clients navigate the API without prior knowledge of its structure. 


{
  "id": 123,
  "title": "Introduction to REST API Design",
  "content": "...",
  "_links": {
    "self": {
      "href": "/articles/123"
    },
    "comments": {
      "href": "/articles/123/comments"
    },
    "author": {
      "href": "/users/456"
    },
    "related": [
      {
        "href": "/articles/124",
        "title": "Advanced REST API Design"
      }
    ]
  }
}

Security Best Practices

Security should never be an afterthought. Here are some essential practices:

  • Use HTTPS: Always secure your API with TLS/SSL
  • Implement proper authentication: Use OAuth 2.0, API keys, or JWT
  • Apply rate limiting: Protect against abuse and DoS attacks
  • Validate and sanitize inputs: Prevent injection attacks
  • Apply appropriate CORS headers: Control access to your API from browsers

Documentation Is Crucial

Even the best-designed API is difficult to use without proper documentation. Consider implementing:

  • Comprehensive API reference (e.g., using Swagger/OpenAPI)
  • Code examples in multiple languages
  • Getting started guides
  • Authentication instructions
  • Detailed error documentation

Conclusion

Designing RESTful APIs with these best practices will make your API more intuitive, consistent, and developer-friendly. Remember that a good API is one that developers enjoy using because it's predictable, well-documented, and follows established patterns.

No API design is perfect from the start—it's an iterative process. Listen to feedback from your API consumers and be prepared to evolve your design over time while maintaining backward compatibility.

Happy API designing!
